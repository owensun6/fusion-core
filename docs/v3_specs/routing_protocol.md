# Fusion-Core V3 协议：跨模型路由 (Cost-Aware Model Routing)

> **目标受众**: 负责维护 `.claude/hooks` 与核心脚本架构的工程兵种。

## 1. 协议目标

大型复杂的系统架构对大模型的逻辑推理能力要求极高，但单纯的函数修补或格式化则纯属浪费算力。需要基于当前的 Stage，动态向 LLM 网关注入特定的 `MODEL_ID` 以实现成本控制 (如最高降本 80%)。

## 2. 路由矩阵 (Routing Matrix)

当 `fusion-router` 被激活截获兵种出动请求时，根据传入的 `ROLE` 或对应的 `STAGE` 自动切换模型。

| 发出请求的兵种 (Role) | 对应的上下文厚度   | 允许的模型选择策略    | 判定理由                                         |
| :-------------------- | :----------------- | :-------------------- | :----------------------------------------------- |
| `pm`, `lead`          | 极厚 (全局+文档)   | **Opus** / **Sonnet** | 架构级思考需要最强的深呼吸与第一性原理推导能力   |
| `db-schema-designer`  | 厚 (全套关联表)    | **Sonnet**            | 数据库设计属于不可逆的核心结构                   |
| `qa-*`, `iv-*`        | 中 (读代码/发报告) | **Sonnet**            | 测试审查涉及安全与长代码段检索                   |
| `fe-ui-builder`       | 薄 (单页面画皮)    | **Haiku** / 小模型    | 无需宏观思考，纯体力搬运 HTML/CSS                |
| `fe-logic-binder`     | 薄 (接数据口)      | **Haiku** / 小模型    | 若已有严谨的后端 Swagger，只需用便宜模型写 Fetch |

## 3. 实现要求 (Hook Override)

修改当前框架的预处理钩子（如 `pre-tool-use.js`），在下发 Agent 执行权限时：

1. 检查当前 `monitor.md` 所处 Stage 及传入的角色名称。
2. 查找路由矩阵表，映射到最低可接受的模型级别。
3. 动态配置对应的环境变量 `CLAUDE_MODEL_NAME` (或系统兼容的模型配置入口)。
4. 在 CLI 向人类输出提示："已自动向 `Haiku` 模型降级执行 UI 切图任务，以为您节约约 95% API 成本"。
