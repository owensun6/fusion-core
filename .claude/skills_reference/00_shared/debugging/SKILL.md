# Fusion 系统化调试 (Systematic Debugging)

> 核心铁律：在找到根本原因（Root Cause）之前，禁止尝试任何修复（Fixes）。

## 一、 四阶段调试流程

任何技术问题的处理必须依次经过以下四个阶段，不得跳步：

### Phase 1: 根本原因调查 (Investigation)

1. **精准阅读错误**：不跳过任何堆栈信息，记录文件名、行号、错误码。
2. **稳定复现**：找到最小可复现步骤。如果无法复现，禁止盲目修改代码。
3. **隔离组件**：在多组件系统中，通过增加日志确定数据在哪一层开始断流或变质。
4. **追溯数据流**：bad value 从哪里产生的？谁传进来的？向上回溯直到源头。

### Phase 2: 模式对比 (Pattern Analysis)

1. **寻找工作范例**：找到代码库中功能相似但正常的代码，对比差异。
2. **查阅参考实现**：阅读核心库的文档或原子动作示例，检查是否理解有误。

### Phase 3: 科学假设与验证 (Hypothesis)

1. **单一假设**：明确表达 "我认为 X 是原因，因为 Y"。
2. **最小化测试**：每次只改动一个变量来验证假设。

### Phase 4: 正式实施 (Implementation)

1. **编写失败测试**：先写一个能触发该漏洞的 Test。
2. **实施最小修复**：只针对根因修复，禁止顺带重构。
3. **验证闭环**：确认测试通过，且没有引入新 Bug。

---

## 二、 危险红灯 (Red Flags)

如果你发现自己有以下念头，请立即停止（STOP）并回到 Phase 1：

- "先试着改一下这里看看行不行" (盲目尝试)
- "这个问题太简单了，不需要走流程" (傲慢)
- "已经试了 3 种修复都没用" (架构性问题，请上报 Commander)
- "我没理解这个错误，但我觉得这行代码有问题" (猜谜)
