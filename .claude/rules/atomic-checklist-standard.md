# 原子级阶段 Checklist 规范

> **Author: Lead**
> 本文件定义 Fusion-Core pipeline 中每个 Stage 内部原子级 Checklist 的格式、判定标准和反模式约束。

---

## 0. 第一性原理强制要求 (First Principles Mandate)

> **[!] CRITICAL**: 这是所有 Checklist 的元规则。违反此规则等于整个阶段作废。

### 0.1 Purpose Anchor（目的锚点）

**每个阶段的 CHECKLIST.md 第一行必须是 Purpose Anchor**，格式如下：

```markdown
> **Purpose**: [一句话] — 这个阶段根本要解决什么问题？
> **如果跳过此阶段**: [具体的、不可接受的后果]
```

**Purpose Anchor 的作用**：所有原子步骤必须能回答"删掉我，Purpose 还能达成吗？"

- 能达成 → 该步骤是冗余的，**删除**
- 不能达成 → 该步骤是必须的，**保留**

### 0.2 执行前两问（前置第一性原理过滤）

执行角色在开始本阶段**任何动作之前**，必须先回答：

```
问题一：我们的目的是什么？
答：[对照 Purpose Anchor 用自己的话复述]

问题二：这些步骤已经不可原子级再分了吗？
答：[逐步检查，发现可再分的立即拆分]
```

这两个问题的答案**不需要写入文件**，但必须在执行角色的内部推理中完成。跳过此步骤等同于违反执行纪律。

### 0.3 计划层 FP 要求（任务拆解时适用）

Lead 在 Stage 3 拆解 task.md 时，每一个 Task 条目必须通过同样的过滤：

- 该 Task 的存在理由是什么？
- 删掉它，上游 Purpose 还能达成吗？
- 它的粒度已经不可再分了吗（2-5分钟内可完成）？

通不过过滤的 Task 条目，**不得写入 task.md**。

## 1. 原子性定义

**一个原子步骤 = 一个可验证的动作 + 一个明确的产出物**

判定法：

- 该步骤能否被**单一命令**或**单一工具调用**完成？
- 如果需要两个不同工具或两个不同认知动作 → **必须拆分**
- 如果产出物无法被第三方独立验证 → **不是原子步骤，需要细化**

---

## 2. Checklist 条目格式

```
- [ ] X.Y 「动作动词 + 对象」: 产出物 → 验证方式
```

| 字段            | 说明                   | 示例                    |
| --------------- | ---------------------- | ----------------------- |
| `X.Y`           | 阶段号.步骤序号        | `0.3`                   |
| 动作动词 + 对象 | 明确的单一动作         | `读取 ROADMAP.md`       |
| 产出物          | 该步骤完成后的可见输出 | `对齐判断 (是/否/偏离)` |
| 验证方式        | 如何确认该步骤确实完成 | `文件存在且非空`        |

---

## 3. 文件结构

每个 CHECKLIST.md 分为两区：

```markdown
<!-- Author: [角色] -->

## 固定骨架 (Standard Steps)

> 每次执行本阶段都必须完成的标准步骤

- [ ] X.1 ...
- [ ] X.2 ...

## 动态扩展 (Project-Specific)

> 执行角色根据项目复杂度在此添加额外步骤
> 格式必须与固定骨架一致

- [ ] (由执行角色动态填写)
```

## 4. 反模式 vs 正模式

### 反模式: 步骤太粗 (WRONG)

```
- [ ] 0.1 完成需求分析: PRD.md → 文件存在
```

**问题**: "完成需求分析"包含了至少 5 个子动作（读取上下文、提问澄清、萃取意图、编写文档、自检一致性），违反原子性。

### 正模式: 每步一个动作 (CORRECT)

```
- [ ] 0.1 读取 ROADMAP.md: 确认需求与战略对齐 → 输出：对齐判断 (是/否/偏离)
- [ ] 0.2 创建 DISCUSSION_LOG.md: 建立讨论日志文件 → 验证：文件存在
- [ ] 0.3 选择路径: V4快速 or 苏格拉底多轮 → 输出：路径选择记录
- [ ] 0.4 执行第 1 轮逼问: 提出 3-5 个边界问题 → 输出：Commander 回答记录
- [ ] 0.5 萃取意图: 从 Q&A 中提取结构化需求 → 输出：需求条目列表
- [ ] 0.6 编写 PRD.md: 填充模板所有章节 → 验证：Author stamp + 章节完整
```

### 反模式: 合并不同工具的操作 (WRONG)

```
- [ ] 1.3 设计 API 契约并编写数据模型: API_Contracts.md + Data_Models.md → 文件存在
```

**问题**: "设计 API 契约"和"编写数据模型"是两个独立的认知动作，需要不同的分析视角，必须拆分。

### 反模式: 产出物不可验证 (WRONG)

```
- [ ] 0.4 思考需求边界: 更深的理解 → 无
```

**问题**: "更深的理解"无法被第三方验证。必须转化为可见产出（如：记录到文件中的 Q&A 清单）。

## 5. 条件步骤

当某步骤仅在特定条件下执行时，标注 `(条件)`:

```
- [ ] 0.5 (条件) 执行第 N 轮逼问: 直到无歧义 → 输出：所有 Q&A 追加记录
```

条件必须在步骤描述中明确说明触发条件。

---

## 6. 执行纪律

1. **禁止跳步**: 必须按序号顺序执行，前一步未勾选不得开始后一步
2. **禁止合并**: 即使两步看起来"很快"，也不得合并为一步执行
3. **禁止遗漏勾选**: 执行完毕后必须立即勾选 `[x]`，不得事后补勾
4. **动态步骤约束**: 动态扩展区的步骤必须遵循相同的原子性格式
