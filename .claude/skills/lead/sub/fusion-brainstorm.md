# fusion-brainstorm — 技术方案头脑风暴


---

## ⚡ 执行前 FP 两问（强制）

1. **我们的目的是什么？**
   → 探索多路径，选最优方案
2. **这些步骤已经不可原子级再分了吗？**
   → 先列方案，再对比权衡，再给出推荐。不跳步，不合并。

---

## ⛔ HARD-GATE（物理锁）

```
本技能完成并经 Commander 确认之前，
禁止 Lead 和任何 Dev 特种兵编写任何生产代码。
```

**物理锁触发条件**: 任何"帮我写 XX 功能"类指令 → 强制退回头脑风暴阶段。

## 执行序列

### Step 1: 理解当前约束

从 Gate 1 通过的架构产出物中提取：

- 技术栈决策（来自 ADR）
- 关键性能/安全约束（来自 System_Design）
- 接口契约限制（来自 INTERFACE.md）

### Step 2: 构造 2-3 种实现路径

对每条路径，描述：

- 核心思路（一句话）
- 主要组件/模块
- 估计复杂度（低/中/高）

**路径构造原则**:

- 路径 1: 最简单可行的实现（MVP 思维）
- 路径 2: 更完整但复杂度适中的实现
- 路径 3（可选）: 面向未来扩展的实现

### Step 3: 权衡分析

```markdown
| 维度       | 路径 1 | 路径 2 | 路径 3（可选） |
| ---------- | ------ | ------ | -------------- |
| 实现复杂度 | 低     | 中     | 高             |
| 可维护性   | 中     | 高     | 高             |
| 性能       | 中     | 高     | 高             |
| 开发时间   | 短     | 中     | 长             |
| 技术风险   | 低     | 中     | 高             |
| v2 扩展性  | 差     | 中     | 好             |
```

### Step 4: 给出推荐方案 + 原因

推荐方案必须能回答：

- 删掉这个设计决策，用户核心痛点还能被解决吗？（FP 刀二）
- 这是 v1 真正需要的，还是过度设计？

### Step 5: 产出设计文档

路径: `pipeline/1_architecture/YYYY-MM-DD-[功能名]-design.md`

```markdown
<!-- Author: Lead -->

**关联 F-ID**: F1.1, F2.3

## 背景与约束

[从 Architecture 产出物中提取的相关约束]

## 方案对比

### 路径 1: [方案名]

**核心思路**: [一句话描述]

**实现要点**:

- [要点 1]
- [要点 2]

**风险**: [主要风险]

### 路径 2: [方案名]

[同上格式]

## 权衡矩阵

[见 Step 3 格式]

## 推荐方案

**推荐**: 路径 [X]

**原因**:

2. [理由 2]

**被拒方案**: 路径 [Y] 因为 [原因] 被放弃
```
