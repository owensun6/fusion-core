# fusion-ux-adversarial-review — 原型批判对手审查


---

## ⚡ 执行前 FP 两问（强制）

1. **我们的目的是什么？**
   → 挑战原型的体验盲区
2. **这些步骤已经不可原子级再分了吗？**
   → 每次只质疑一条路径/一个场景，不要一次性爆出所有问题。

---

## 执行序列

### Step 1: 阅读所有产出物（不加评论地阅读完）

读取顺序（重要：全读完再开始审查，不边读边评）：

1. `pipeline/0_requirements/FEATURE_LIST.md` — 了解需要覆盖的 F-ID
2. `pipeline/0_requirements/BDD_Scenarios.md` — 了解需要覆盖的场景
3. `pipeline/0_5_prototype/Feature_Screen_Map.md` — 了解功能→屏幕映射
4. `pipeline/0_5_prototype/User_Flow.md` — 了解已设计的用户路径
5. `pipeline/0_5_prototype/Wireframes/` — 了解各屏幕的设计内容
6. `pipeline/0_5_prototype/UI_CONTRACT.md` — 了解前端约束定义

### Step 2: F-ID 覆盖检查

逐条对比 FEATURE_LIST 中的每个 F-ID 与 Feature_Screen_Map：

| 检查项                                              | 通过条件 |
| --------------------------------------------------- | -------- |
| 每个 F-ID 在 Feature_Screen_Map 中有对应屏幕        | 全覆盖   |
| 每个 F-ID 在 UI_CONTRACT 的覆盖矩阵中有对应组件     | 全覆盖   |
| BDD_Scenarios 中的每个场景在 User_Flow 中有对应路径 | 全覆盖   |

### Step 3: 路径完整性拷问

对 User_Flow 中每条核心路径，主动构造至少 2 个未被设计的异常场景：

| 构造角度   | 示例问题                                         |
| ---------- | ------------------------------------------------ |
| 操作中断   | 用户操作到第2步时关闭浏览器，再次打开会怎样？    |
| 网络失败   | 提交时网络中断，用户会看到什么？数据会丢失吗？   |
| 新手误操作 | 用户在不该点的地方点了，有没有确认弹窗？         |
| 并发场景   | 两个用户同时修改同一条记录，谁的修改会生效？     |
| 空数据状态 | 列表页为空时（无数据），用户看到什么？怎么引导？ |
| 权限边界   | 普通用户访问了只有管理员才能访问的屏幕，会怎样？ |
| 极端数量   | 列表有 10000 条记录，滚动/加载如何处理？         |

### Step 4: UI_CONTRACT 完整性检查

| 检查维度       | 具体问题                                                              |
| -------------- | --------------------------------------------------------------------- |
| 状态覆盖       | 每个组件的所有状态变体（初始/聚焦/加载/成功/失败/禁用）是否都有定义？ |
| 跳转完整性     | 每个操作的所有可能结果（成功/失败/取消）都有对应跳转目标？            |
| 空状态定义     | 所有列表型组件是否有空状态（无数据时）的设计规则？                    |
| 错误文案质量   | 错误提示是用户能理解的语言，还是技术错误码？                          |
| 全局规则一致性 | 同类操作（如删除确认）是否遵循了相同的交互模式？                      |

## 判定规则

| 情况             | 结论               | 后续                         |
| ---------------- | ------------------ | ---------------------------- |
| 存在 CRITICAL    | **REVISE**         | UX Designer 返工，重新提交   |
| 只有 HIGH/MEDIUM | **PASS（附报告）** | 报告随产出物提交 Commander   |
| 无问题           | **PASS**           | 进入 Gate 0.5 Commander 确认 |

**连续 3 次 REVISE → 熔断**: 输出"Commander，已连续 3 轮无法通过审查，触发熔断。是否退回 Stage 0 重新定义需求？"
