## 审查维度

### D1: 业务不变量 (Invariants)

核心业务规则是否在所有代码路径上都得到保护：

**通用不变量示例**:

- 账户余额不可为负 → 扣款前是否有余额检查？
- 订单状态转换必须遵循状态机 → 是否有非法状态转换的防护？
- 库存不可超售 → 是否有并发安全的库存检查？

**验证方法**:

- 找到业务不变量定义（PRD 或 Data_Models）
- 追踪所有可能修改该字段的代码路径
- 确认每条路径都有不变量保护

### D2: 数据完整性约束

数据层约束是否在代码层也有相应校验：

- 外键关联：创建子记录前是否验证父记录存在？
- 唯一约束：代码层是否有唯一性预检（而不是依赖 DB 抛异常）？
- 非空约束：是否在服务层也验证必填字段？

**原则**: DB 约束是最后防线，代码层应该有相应的先行验证。

### D3: 领域规范合规（医疗 IT 场景）

针对医疗 IT 项目的特殊检查：

- **诊断码/药品码**: 是否经过 ICD-10 / ATC 等标准码表校验？
- **患者数据访问控制**: 是否符合等保 2.0 / HIPAA 的访问控制要求？
- **电子病历不可删除原则**: 病历修改是否创建修订记录而非覆盖原始记录？
- **处方权限**: 开具处方操作是否验证医师资质级别？

### D4: 计算精度

精度敏感字段检查：

- 货币金额是否使用 `Decimal` 或整数（分）而非 `float/double`？
- 医疗剂量、浓度、百分比是否有精度定义？
- 浮点数比较是否使用 epsilon 而非直接 `===`？

### D5: 边界条件的业务语义

极限值在业务上的语义是否正确处理：

- 数量为 0 的订单：业务上是否允许？如何处理？
- 最大值（如年龄超过 150）：是否有合理的上限验证？
- 负数输入：业务上是否有意义？代码是否正确拒绝？

## PRD 验收对齐（逐条核查）

读取 `TASK_SPEC_T-{ID}.md` 中的验收标准，逐条用代码实现来对照：

```markdown
## PRD 验收对齐检查

- [ ] 验收标准 1: PASS/FAIL — 说明
      → 对应代码位置: file:line
- [ ] 验收标准 2: PASS/FAIL — 说明
      → 对应代码位置: file:line
```
