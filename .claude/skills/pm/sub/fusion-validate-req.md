# fusion-validate-req — 需求验证与 FP 拷问

> **适用角色**: PM 自检 + PM Consultant 对抗审查（两者均使用此工具，视角不同）

---

## ⚡ 执行前 FP 两问（强制，不可跳过）

1. **我们的目的是什么？**
   → 验证产出物可独立交付 Lead
2. **这些步骤已经不可原子级再分了吗？**
   → 五个维度逐一检查，不可合并，每个维度独立出结论。

---

## 触发条件

| 角色          | 触发时机                                   |
| ------------- | ------------------------------------------ |
| PM 自检       | `fusion-compile-req` 完成后，Gate 0 提交前 |
| PM Consultant | PM 自检通过后，Commander 签字前            |

## 两大核心基准（FP Check 灵魂）

**Commander 原创，每个维度都要用这两把刀去问：**

### 刀一：步骤不可再分（Atomicity）

> 这个需求条目、功能点或 BDD 场景，真的已经不可再切分了吗？
> 是否存在一个 Scenario 其实包含了两个独立的业务事件？

### 刀二：目的是什么（Intent Clarity）

> 这一条的根本动机是什么？是真需求还是过度设计的伪装？
> 如果删掉这一条，用户的核心痛点还能被解决吗？

---

## 五维度审查清单

### 维度 A: 完备性 (Completeness)

- [ ] 每个核心用户角色（非仅管理员）的主流程均有对应场景
- [ ] 每个功能点都有 Happy Path + Error Case，无遗漏
- [ ] 非功能性需求（并发、安全等级）均有描述或明确标注 TBD
- [ ] 系统边界（哪些不做）在 PRD 中明确列出

### 维度 B: 清晰度与可测量性 (Clarity & Measurability)

> 来源：CC-Best requirement-validator（补充）

- [ ] 无模糊词汇（"快速""简单""大量""好用"）— 发现即标记，要求量化
- [ ] 所有数值指标具体可测（❌"响应快" ✅"响应时间 < 200ms"）
- [ ] 每条验收标准可以直接写出测试用例（能否说出"Given...When...Then..."）
- [ ] 成功指标有数值，可以客观判断"完成"（避免"差不多就行"）
- [ ] 术语定义清晰，全文一致无歧义

### 维度 C: 一致性 (Consistency)

- [ ] PRD 用例 ↔ FEATURE_LIST 功能点 ↔ BDD 场景 三者一一对应，无孤立条目
- [ ] 所有业务实体（如"处方单""患者""库存"）在全文统一命名，无歧义
- [ ] 置信度"低"的假设已被 Commander 确认或调整

### 维度 D: 合理性 (Reasonableness)

用两大核心基准拷问每个功能点：

- [ ] 每条功能点通过了"目的是什么"测试（删掉它，痛点还在吗？）
- [ ] 每条 BDD Scenario 通过了"不可再分"测试（一个 Scenario 只测一件事）
- [ ] 无过度设计（v1 不需要的复杂性）

### 维度 E: 风险盲区 (Blind Spots)

以 **Critical Adversary（批判对手）** 视角主动构造以下场景：

- [ ] 恶意用户 / 操作失误的边缘场景有对应 Error Case
- [ ] 并发操作（两人同时操作同一数据）的竞争条件有说明
- [ ] 所有数字（用户量、数据量、频率）有来源依据或明确标注 TBD
- [ ] 至少 1 个 PM 原文中**未覆盖的边缘场景**被识别并处理

### 维度 F: BDD 质量 (BDD Quality)

- [ ] Gherkin 语法正确（Feature / Scenario / Given / When / Then）
- [ ] Given 包含具体数据（不是"某个用户"，是"拥有处方权的医生 Doc_001"）
- [ ] Then 描述可观测行为（不是"成功"，是"用户看到X提示并跳转到Y"）
- [ ] 无技术术语（无 SQL、API 路径、字段名）

## 判定规则

| 情况               | 结论               | 后续                                    |
| ------------------ | ------------------ | --------------------------------------- |
| 存在 CRITICAL 问题 | **REVISE**         | PM 必须修复后重新提交，重新执行本技能   |
| 只有 HIGH/MEDIUM   | **PASS（附报告）** | 附报告提交 Commander，由 Commander 决定 |
| 无问题             | **PASS**           | 进入 Gate 0 等待 Commander 签字         |

---

## 连续 3 次 REVISE 熔断机制

如果同一份产出物被连续 REVISE 3 次：

1. 停止技术尝试
2. 输出："Commander，已连续 3 轮无法通过验证，触发熔断机制。是否需要退回 Stage 0 重新定义需求？"
3. 等待 Commander 决定，系统挂起
