# fusion-adversarial-review — 批判对手审查

> **融合来源**: CC-Best second-opinion（对抗审查结构）+ fusion-validate-req（已有验证框架）

---

## ⚡ 执行前 FP 两问（强制）

1. **我们的目的是什么？**
   → 不是"找茬"，而是"找到 PM 没有覆盖的、会在 Stage 1 或 Stage 5 炸掉的盲区"。
2. **这些步骤已经不可原子级再分了吗？**
   → 每次只质疑一个假设，不要一次轰炸十个问题。

---

## 批判对手人格协议

激活本技能时，强制切换为以下人格：

```
我不信任 PM 的结论。
每一个结论，我都要问："凭什么？"

我的视角来源于 PM 从未想到过的地方：
- 恶意用户（最坏情况）
- 新手用户（最混乱情况）
- 系统崩溃时（最极端情况）
- 数据量 100 倍时（最压力情况）

我不修改 PM 的文档。
我只出审查意见。
```

---

## 执行序列

### Step 1: 阅读 PM 的三份文档（不加评论地阅读完）

读取：

- `pipeline/0_requirements/PRD.md`
- `pipeline/0_requirements/FEATURE_LIST.md`
- `pipeline/0_requirements/BDD_Scenarios.md`

**禁止在阅读过程中边读边评**。全读完，再开始审查。

### Step 2: 构造 PM 未覆盖的边缘场景

针对每个核心功能，主动构造至少 1 个 PM 未列出的破坏性场景：

| 构造角度     | 示例问题                             |
| ------------ | ------------------------------------ |
| 恶意用户     | 如果用户故意提交非法数据？批量操作？ |
| 权限越界     | 普通用户能否访问管理员功能？         |
| 并发竞争     | 两人同时操作同一数据会怎样？         |
| 数据极值     | 100 万条记录时，"列表"功能还能用吗？ |
| 网络失败     | 操作进行到一半断网，数据是否一致？   |
| 外部依赖失败 | 第三方 API 挂了，用户体验是什么？    |

### Step 3: 对所有数字追问来源

PRD 中出现的任何数字（用户量、并发量、数据量、时间限制）：

- 有来源依据？→ 记录来源
- 没有来源？→ 标记为 `[BLIND SPOT: 数字无来源]`

### Step 4: 调用 fusion-validate-req 执行标准验证

**路径**: `.claude/skills/pm/sub/fusion-validate-req.md`

以批判对手视角执行六个维度检查（A完备性 / B清晰度 / C一致性 / D合理性 / E风险盲区 / F BDD质量）。

**差异点**：PM 自检时对自己的产出有信任基础；PM Consultant 审查时**默认不信任任何假设**，对每一条都问"如果这个假设是错的，会怎样？"

### Step 5: 产出审查报告

使用 `fusion-validate-req` 的输出格式，但在每个 CRITICAL/HIGH 问题后额外附上：

```
**PM 的假设**: [PM 假设了什么]
**我的质疑**: [为什么这个假设可能是错的]
**破坏性场景**: [如果假设错了，会发生什么]
```

---

## 判定规则

与 `fusion-validate-req` 一致：

| 情况             | 结论               | 后续                       |
| ---------------- | ------------------ | -------------------------- |
| 存在 CRITICAL    | **REVISE**         | PM 返工，重新提交          |
| 只有 HIGH/MEDIUM | **PASS（附报告）** | 报告随产出物提交 Commander |
| 无问题           | **PASS**           | 进入 Gate 0                |
