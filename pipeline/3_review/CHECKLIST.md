# Stage 6: 代码审查与集成测试 — 原子级 Checklist

> **Purpose**: 在代码合并主分支之前，通过 7 道独立审查漏斗，确保代码质量、安全性、契约合规性和端到端功能全部达标。
> **如果跳过此阶段**: 带 Bug 和安全漏洞的代码进入主分支，线上事故成本远高于此阶段的审查成本。
> **执行角色**: Reviewer（qa-01 → qa-02 → qa-03 → iv-01 → iv-02 → iv-03，串行管道，前一道不过后一道不开）
> **规范参考**: `.claude/rules/atomic-checklist-standard.md`, `.claude/rules/integration-tests-checklist.md`
> **Gate**: Gate 3 (Commander 签字)

---

## 追溯链 (Traceability Chain)

```
Stage 5 产出物（src/ + tests/）
  → qa-01 代码语法与逻辑 ← 单元测试覆盖率
    → qa-02 规范合规 ← 契约对齐 INTERFACE.md
      → qa-03 安全审计 ← OWASP Top 8
        → iv-01 端到端连通 ← E2E 主流程
          → iv-02 数据穿透 ← 数据落盘验证
            → iv-03 混沌测试 ← 边界值与极端场景
              → Audit_Report.md + Integration_Report.md
                → Gate 3 (Commander 签字)
```

**铁律**: 串行管道，前一道审查结论为 FAIL 时，后续道次不得启动。立即打回 Stage 5 对应兵种返工。

---

## Phase A: 代码质量审查（qa-01 → qa-03）

### qa-01: 代码语法与逻辑

- [ ] 6.1 确认所有 Stage 5 任务 Worker 状态均为 `[x]`: 读取 `pipeline/monitor.md` → 验证：无 `[ ]` 状态的任务
- [ ] 6.2 运行单元测试套件: 执行 `npm test` 或等效命令 → 输出：测试结果（通过率 + 覆盖率数字）
- [ ] 6.3 验证覆盖率 ≥ 80%: 检查行覆盖率、分支覆盖率 → 低于 80% 立即标记 FAIL，打回对应 Dev 补充测试
- [ ] 6.4 静态扫描逻辑错误: 审查空指针、死循环、边界值处理、未处理 Promise rejection → 输出：逻辑错误清单（如无则记录"无发现"）

### qa-02: 规范合规

- [ ] 6.5 逐接口核查 INTERFACE.md 对齐: 检查每个实际实现的接口与 INTERFACE.md 定义的端点/参数/响应格式是否完全一致 → 输出：不一致项清单（如无则记录"全部对齐"）
- [ ] 6.6 验证入参校验覆盖: 检查每个接口是否有完整的 Zod/Joi Schema 校验 → 输出：校验缺失清单
- [ ] 6.7 检查错误码标准化: 验证所有错误响应格式符合 `patterns.md` 中定义的统一信封格式 → 输出：不规范项清单

### qa-03: 安全审计

- [ ] 6.8 扫描硬编码密钥: 检查代码中是否存在硬编码的 API Key、密码、Token → 发现立即 STOP，标记 FAIL
- [ ] 6.9 验证 SQL/NoSQL 注入防护: 检查所有数据库查询使用参数化方式，无字符串拼接 SQL → 输出：风险点清单
- [ ] 6.10 检查越权漏洞 (IDOR): 验证每个接口的权限校验逻辑，用户只能访问自己的资源 → 输出：权限检查结果
- [ ] 6.11 验证前端渲染安全: 检查前端无不安全的 HTML 直接注入（需使用经过净化的渲染方式）→ 输出：渲染安全风险清单
- [ ] 6.12 检查速率限制: 验证公网暴露的高成本接口和鉴权接口有速率限制配置 → 输出：缺失速率限制的接口清单

### Phase A 自检

```
- [ ] qa-01 结论：PASS（覆盖率 ≥ 80%，无逻辑错误）
- [ ] qa-02 结论：PASS（接口全部对齐，入参校验完整）
- [ ] qa-03 结论：PASS（无硬编码密钥，无注入风险，权限校验完整）
- [ ] 三道全部 PASS 方可进入 Phase B，任意一道 FAIL 立即打回 Stage 5
```

## Phase B: 集成验证（iv-01 → iv-03）

### iv-01: 端到端连通性

- [ ] 6.13 执行 E2E 主流程测试: 使用 Playwright 或等效工具，跑通所有 BDD 正常流 Scenario → 输出：E2E 测试结果（通过数/总数）
- [ ] 6.14 验证 HTTP 状态码全绿: 核心流程无 500/400/404 → 输出：状态码异常清单（如无则记录"全绿"）
- [ ] 6.15 验证 CORS 配置正确: 跨域请求不被错误拦截 → 输出：CORS 测试结果

### iv-02: 数据穿透性

- [ ] 6.16 验证数据落盘完整性: UI 层提交的数据能准确写入 DB，读回时字段无丢失、无类型转换异常 → 输出：数据穿透测试结果
- [ ] 6.17 验证并发写入保护: 模拟并发写入同一资源，乐观锁/悲观锁机制生效 → 输出：并发测试结果
- [ ] 6.18 (条件) 验证缓存失效机制: 若项目有缓存层，数据更新后缓存正确失效 → 输出：缓存失效测试结果

### iv-03: 混沌与边界

- [ ] 6.19 注入边界值: 空字符串、超大数字、特殊字符、null → 系统不崩溃，返回友好错误 → 输出：边界值测试结果
- [ ] 6.20 模拟超时场景: 触发长事务超时，验证优雅降级（不是 500 崩溃）→ 输出：超时降级测试结果
- [ ] 6.21 执行 BDD 异常流 Scenario: 确认所有 BDD 异常场景都有正确的系统响应 → 输出：异常流覆盖报告

### Phase B 自检

```
- [ ] iv-01 结论：PASS（E2E 主流程全通，无 5xx 错误）
- [ ] iv-02 结论：PASS（数据落盘完整，并发保护生效）
- [ ] iv-03 结论：PASS（边界值不崩溃，异常流全覆盖）
- [ ] 三道全部 PASS 方可进入 Phase C
```
